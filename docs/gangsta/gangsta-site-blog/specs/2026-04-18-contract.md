# Contract: Gangsta Site Blog

## Objective
Add a blog section to the existing gangsta.page Nuxt 4 / Docus v5 documentation site. The blog must be completely separate from the documentation sidebar navigation, accessible via a dedicated `/blog` route that lists all articles, with individual articles rendered in a simple layout (no docs sidebar, no TOC, no breadcrumbs). A "Blog" link must appear in both the site header and the site footer.

## Requirements

### Functional Requirements

1. **FR-001** — A `/blog` route must exist and render a list of all published blog articles, sorted by date descending.
2. **FR-002** — Each article in the list must display: title, description, and date.
3. **FR-003** — Each article must be linkable from the list to its individual page at `/blog/<slug>`.
4. **FR-004** — Individual blog post pages must render the full markdown content of the article.
5. **FR-005** — A "Blog" link must appear in the site header navigation alongside existing header elements.
6. **FR-006** — A "Blog" link must appear in the site footer.
7. **FR-007** — The blog section must NOT appear in the documentation sidebar navigation.
8. **FR-008** — Navigating to a non-existent blog slug must result in a proper 404 response, not a blank or broken page.
9. **FR-009** — At least two sample blog posts must exist as content, to validate the listing and routing.

### Non-Functional Requirements

1. **NFR-001** — Blog pages must be visually consistent with the existing site design system: Nuxt UI components, Tailwind 4, amber primary color.
2. **NFR-002** — Blog post markdown frontmatter must support at minimum: `title`, `description`, `date`. No other required fields.
3. **NFR-003** — Blog post filenames and URL slugs must be prefixed with the publication date in `YYYY-MM-DD` format (e.g., `content/blog/2026-04-18-my-first-post.md` → `/blog/2026-04-18-my-first-post`). This enforces chronological ordering by filename.
3. **NFR-003** — The blog must be statically generated (SSG-compatible) — no server-side-only APIs.
4. **NFR-004** — No new runtime dependencies may be introduced without Don approval.
5. **NFR-005** — The existing documentation routes must continue to function without regression.

## Architectural Decisions

### Decision 1: Custom Nuxt Pages for Blog Routing
Blog routing is handled by `app/pages/blog/index.vue` (list) and `app/pages/blog/[...slug].vue` (individual post). Custom pages in `app/pages/` take routing priority over the Docus content catch-all in the layer.

**Rationale:** Individual blog posts require a simple layout without the Docus docs chrome (sidebar, TOC, breadcrumbs). A custom page is the only way to achieve this layout independence.

### Decision 2: Path-Based Content Query — No `content.config.ts`
Blog posts are stored in `content/blog/` and queried using a path-based filter on the default Nuxt Content collection. No project-level `content.config.ts` is introduced.

**Rationale:** Docus v5 provides its own Nuxt Content layer configuration. Introducing a project-level `content.config.ts` risks silently breaking the docs collection due to undocumented merge behavior. Path-based querying is safe and sufficient.

### Decision 3: Custom Vue Components for Header and Footer Links
The "Blog" link in the header is added via a custom component that overrides the Docus `AppHeader` slot (or extends `AppHeaderLogo`). The footer link is added via the existing `AppFooterLeft.vue` custom component.

**Rationale:** The Docus `app.config.ts` `footer.credits` is a single-object config, not a links array. Arbitrary navigation links cannot be injected via config alone.

### Decision 4: `navigation: false` in Blog Content Directory
`content/blog/.navigation.yml` sets `navigation: false` to exclude the blog directory from the Docus sidebar navigation tree entirely.

**Rationale:** Docus automatically includes all `content/` subdirectories in the sidebar. The `navigation: false` flag is the established pattern (already used in `content/index.md`).

## Grilling Conclusions

### Key Decisions
- **Path-based query over named collection**: Avoids Docus layer content config conflict
- **Custom `[...slug].vue` page**: Required to achieve simple layout without docs chrome
- **Custom Vue components for header/footer**: `app.config.ts` does not support arbitrary link injection in the relevant slots
- **404 guard in slug page**: Null content check with explicit `throw createError({ statusCode: 404 })` prevents broken renders

### Rejected Alternatives
- **Named `blog` collection via `content.config.ts`**: Rejected — undocumented merge behavior with Docus layer config poses high regression risk to existing docs
- **Docus docs layout for blog posts**: Rejected — Don explicitly requested simpler layout without sidebar/TOC/breadcrumbs
- **External blog platform (Ghost, Substack)**: Rejected — out of scope per Don's intent

### Unresolved Objections
- None — all Devil's Advocate attacks resolved

## Acceptance Criteria

1. Navigating to `/blog` renders a list of at least 2 articles with title, description, and date visible.
2. Clicking an article navigates to `/blog/YYYY-MM-DD-<slug>` and renders the full markdown content.
3. Article filenames follow the `YYYY-MM-DD-slug.md` pattern.
3. The "Blog" link is present and functional in the site header.
4. The "Blog" link is present and functional in the site footer.
5. The blog directory does NOT appear as a section in the documentation sidebar.
6. Navigating to `/blog/nonexistent-slug` returns a 404 page.
7. All existing documentation routes (`/getting-started/*`, `/core-concepts/*`, etc.) continue to render correctly.
8. The site builds without errors via `bun run build`.

## Out of Scope
- Blog search functionality
- RSS/Atom feed
- Pagination on the blog list page
- Tags, categories, or author pages
- Comments system
- Blog-specific SEO beyond basic frontmatter-derived meta tags
- CMS or admin interface for writing posts

## Open Risks
- None from Grilling (all resolved). The content query approach should be validated early during implementation against the actual Nuxt Content v3 API available in the Docus v5 layer.
